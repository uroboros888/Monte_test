## Цели
- Обеспечить единообразное описание экспериментов и сценариев с помощью читаемого DSL.
- Автоматизировать конвейер подготовки данных, запуска моделирования и агрегации результатов.
- Сохранять результаты в структурированном JSON-формате для интеграции с внешними системами и аналитикой.
- Гарантировать расширяемость ядра за счёт модульной архитектуры и слабой связности компонентов.

## Конвейер обработки
| Этап | Вход | Выход | Основные компоненты |
|------|------|-------|---------------------|
| **1. Подготовка входных данных** | DSL-описания, внешние источники | Нормализованные артефакты | Loader, Data Normalizer |
| **2. Парсинг DSL** | DSL-файлы | Промежуточная модель сценария | Parser, Validator |
| **3. Генерация JSON-конфигурации** | Модель сценария | `simulation.json` | JSON Generator, Schema Registry |
| **4. Выполнение ядра** | JSON-конфигурация | Результаты моделирования | Execution Engine, Task Scheduler |
| **5. Постобработка** | Результаты моделирования | Агрегированные метрики | Result Aggregator, Metrics Service |
| **6. Экспорт** | Агрегированные метрики | Отчёты, артефакты | Export Service, Integrations |

> **Примечание:** Все этапы публикуют события в шину телеметрии; это позволяет строить мониторинг и реконструировать ход выполнения.

Последовательность выполнения управляется оркестратором: при необходимости отдельные этапы могут быть включены или отключены флагами конфигурации.

## Компонентная структура
- **Интерфейс DSL** — набор библиотек для редактирования и статической проверки сценариев, включает CLI и редакторские шаблоны.
- **Оркестратор** — отвечает за планирование выполнения конвейера, управляет зависимостями и состоянием задач.
- **Исполнительные адаптеры** — изолируют ядро от конкретных вычислительных платформ (локальный режим, Kubernetes, облачные сервисы).
- **Служба данных** — предоставляет доступ к историческим результатам, артефактам и конфигурациям запусков.
- **Телеметрия и наблюдаемость** — собирают метрики, логи и трассировки; интегрируются с системами оповещения.

## Структура данных и контракты
Основным артефактом является JSON-конфигурация запуска. Ниже приведён укрупнённый пример:

```json
{
  "metadata": {
    "schema_version": "1.2",
    "run_id": "2024-05-18T12:00:00Z",
    "initiator": "ci-pipeline"
  },
  "inputs": {
    "datasets": ["market.csv", "demand.csv"],
    "parameters": {"horizon": 30, "seed": 42}
  },
  "pipeline": ["normalize", "simulate", "aggregate"],
  "results": {},
  "artifacts": [],
  "extensions": {}
}
```

Требования к данным фиксируются в **Schema Registry**. Для каждого типа артефакта хранится версия схемы и миграции между версиями. Изменение структуры требует:

1. Обновления схемы в реестре.
2. Реализации миграции исторических данных (при необходимости).
3. Обновления документации DSL.

## DSL
- Основан на декларативном описании сущностей: сценарии, источники данных, политики моделирования и метрики.
- Поддерживает макросы (`include`, `template`) для переиспользования конфигураций.
- Валидация выполняется парсером на основе схем, синхронизированных с JSON-структурой.
- DSL служит единственным источником правды: любые изменения сценариев фиксируются в системе контроля версий.

Пример DSL-фрагмента:

```dsl
scenario "baseline" {
  dataset "market" from "s3://bucket/market.csv"
  parameter horizon = 30

  task simulate {
    engine = "monte"
    seed = 42
  }

  metric "roi" using aggregator.percentile(0.9)
}
```

## Ядро исполнения
- **Execution Engine** — управляет жизненным циклом задач, реализует ретраи и таймауты.
- **Task Scheduler** — распределяет задачи по доступным исполнителям и обеспечивает балансировку нагрузки.
- **Plugin System** — расширяет функциональность ядра новыми типами задач и обработчиков результатов.
- **Observability Hooks** — собирают телеметрию (метрики, логи, трассировки) и передают её в централизованные системы.

## Расширяемость и интеграции
- Новые этапы конвейера регистрируются через декларативный механизм и автоматически появляются в оркестраторе.
- JSON-схема версионируется; поддерживаются параллельные схемы для A/B тестирования.
- Предусмотрены интеграции с системами оркестрации (Airflow, Argo), CI/CD и BI-платформами.
- Через API-плагины можно подключать внешние сервисы расчётов (например, ML-облако).

## Тестирование и контроль качества
- **Статическая валидация** — проверка DSL и JSON на соответствие схемам до запуска моделирования.
- **Модульные тесты** — покрывают парсер, генератор JSON и плагины ядра.
- **Интеграционные тесты** — эмулируют полный цикл конвейера на сокращённых наборах данных.
- **Наблюдаемость в продакшене** — алерты по SLA этапов и автоматическое открытие инцидентов.

## План развития
- Добавление визуального редактора сценариев с поддержкой автодополнения.
- Расширение набора стандартных плагинов (имитационное моделирование, оптимизационные задачи).
- Введение автоматических миграций при изменении схемы данных.
- Интеграция с системой экспериментов для сравнения результатов запусков.
